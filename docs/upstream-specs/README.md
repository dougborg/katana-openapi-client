# Upstream OpenAPI specs

Two complementary upstream sources, normalized to YAML for diff-friendly storage.
Everything in this directory is **auto-generated by** `poe refresh-upstream-spec`
(script: [`scripts/pull_upstream_specs.py`](../../scripts/pull_upstream_specs.py)). **Do
not hand-edit.** Refresh by re-running the task.

## Files

### [`live-gateway.yaml`](./live-gateway.yaml)

The canonical OpenAPI spec served by the live API gateway at
`https://api.katanamrp.com/v1/openapi.json`. Public endpoint, no auth. This is what the
gateway *actually validates requests against*.

- **Use it for**: request DTOs, paths, filter-param enums, component schemas. This is
  the authoritative source for *what the API accepts*.
- **It does not have**: response shapes. Every `200` response is just
  `{description: "Return value of FooController..."}` with no schema or example.

This is what `poe audit-spec` compares the local spec against.

### [`readme-portal.yaml`](./readme-portal.yaml)

The OpenAPI spec embedded in any reference page on `https://developer.katanamrp.com/` —
lifted out of `<script id="ssr-props">` at `document.api.schema`. The pull script
derives the page slug at runtime via the README.io Algolia search index, so it doesn't
depend on any specific endpoint slug being stable.

- **Use it for**: response examples. Every endpoint's
  `responses.<status>.content.<ctype>.example` carries a real JSON payload curated by
  Katana's docs team. This is the primary source `poe validate-response-examples`
  validates our local response schemas against.
- **It is slightly behind** the live gateway on path coverage. Treat request-side info
  here as advisory only — `live-gateway.yaml` wins for request DTOs.

## Refresh

```bash
poe refresh-upstream-spec
```

Idempotent. Both fetches run in parallel via `asyncio.gather`. Output is written through
`yaml.safe_dump(..., sort_keys=True)` so diffs are stable across runs.

## Why YAML?

JSON would also work, but:

- YAML is quote-free / comma-free → easier to read in PR review.
- `sort_keys=True` gives stable diffs across upstream re-orderings.
- The same formatter is used for both files, so they diff cleanly against each other
  (e.g. *which paths are in `live-gateway` but not in `readme-portal`?*) and against our
  local spec at [`../katana-openapi.yaml`](../katana-openapi.yaml).

## Why two specs and not one?

Katana publishes the OpenAPI spec in two places, each with different content:

- The **live gateway** ships the spec the runtime validator uses — request DTOs and
  filter enums — but no response schemas or examples (the gateway just doesn't carry
  response info in its OpenAPI doc).
- The **README.io developer portal** ships a spec with inline response examples curated
  by humans for the docs site. It's slightly behind on request-side coverage but it's
  the only structured place response examples exist.

Each spec is the source of truth for a different part of our local spec. Pulling both
keeps our audit tooling honest in both directions.

## History

The legacy `docs/katana-api-comprehensive/` directory contained scraped markdown for
every endpoint plus an embedded OpenAPI JSON. The scrape broke when README.io migrated
to a JS-rendered SPA, and the markdown parsing was always brittle. This directory
replaces that legacy archive with structured, refreshable, normalized YAML.
